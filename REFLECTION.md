# 总况
通过这个项目的训练，我对 Agent coding 有了更多的经验和更深入的理解。主要有以下几点收获：
1. Superpowers 流程对于新项目开发并非必要，至少很多地方可以按需精简。
2. 快速构建 MVP ，根据遇到的问题进行重构，是比一个流程顺下来更好的方式。
3. 时不时观察Agent在做什么是有必要的，可以省去很多麻烦。

## 1. Superpowers: Really Useful?

先从总体上的数据来看，引入superpowers后的工作流中，审计性文字和代码输出的量近乎为1:1， 这是一个极其不合理的数字。仔细观察可以看到，有时候存在滥用 Subagent 的迹象。很多时候，很小的修复，也必须按照流程启动subagent进行复杂的全量构建。

从实际的 coding 感受来看，spec 文档有时候并不是必要的，尤其是在初版开发完成、后续添加功能和修复bug的时候。有多次这样的情况：我只是提出了一个很简单的需求，却需要多次反复确认；甚至后面都已经完全疲于审核而直接回复“是的”、“继续”了。这就像去楼下便利店买东西还要开车一样。worktree的建立也有待审视，特别是“每个task都建立一个worktree”这样的设计。我在实施前两个task的时候就意识到这种设计过于低效——很多task完全可以而且有必要在一个worktree上实现。

事实上，我前后进行了两版开发，有一个开发到一半舍弃了，就是因为superpowers造成了过于臃肿的项目结构。后面，我将superpowers从全局skill中移除，只是放在项目的文件夹中，在AGENT.md中说明要使用——这样的话其实模型不会强制走这个流程，会更遵循我的指令。

还有一个很重要的缺陷——太费token了！以往我做这种量级的项目，总共可能也就用 codex 20%~30%的周限额（5xpro套餐），然而这个一下子用掉了我 180%的额度。虽然也有 5.6 价格提升的缘故，但是从其skill本身就消耗巨量token来看，superpowers称为“token粉碎机”也不为过。

然而superpowers却也是有用的。我认为几个比较有用的点有：
1. brainstorm。如果没有这个，你很难在项目一开始表述清楚自己的需求。并且这个在后续的功能添加中也会用到，具体体现是 Agent 会让你做选择——做选择是必然比凭空写需求要简单的多的。不过，这个功能对于 claude code 来说其实帮助有限，因为 claude code （使用claude模型时）本来就会让用户做一些选择。对于 codex 的帮助很大，因为以往使用codex的时候，总是出现的情况是我提出一些需求之后，他就吭哧吭哧干了，有时候不免会出现理解偏差而已经干完了的情况——因此网上也会出现一些“claude擅长构建而codex擅长执行”的说法。我认为这个skill，或者至少其核心思想，是可以在以后的workflow中保留的。
2. TDD、review等机制，确实在未人工测试之前就暴露出很多问题，最终的产物几乎是不需要大改的。

总而言之，我认为superpowers固然有用，但是有过度设计的嫌疑。在不考虑token消耗、项目刚刚开始的阶段，确实可以作为一个拿来直接用的框架。但是并不适用于所有项目、所有开发细节。更好的实践，应该结合自己具体的需求来简化——这样也能避免人过于惰性的 vibe coding。

## 2. 重构的必要性

按照常理来讲，我们很多时候是不希望重构的，特别是产品已经有很多人使用的时候。但是在项目最初版的开发阶段，我认为适时的进行重构还是有必要的。主要原因在于，在当下我们已经没有时间从头仔细review每一行代码，只能保证大体功能上的正确性。这样一来，出问题的时候，只能靠Agent定位问题并进行修复，从而导致技术债逐渐累积。如果在问题较多的时候进行适时的重构，我们是带着实践中得到的经验的，在架构的选择、逻辑的设计等方面都会更有目的性。而且，做好重构的准备，也能让我们在构建初版的时候以更加开放的心态大胆尝试、快速迭代，其实反而提升了效率。

## 3. 做一个合格的监工

Agent coding的一大优势是我们可以让它自己跑，可以同时做很多事情。但是问题也往往出在这里。即使是经过 Brainstorm，agent也有可能会有一些明显不符合预期的行为。比如我这次实践过程中，很明显的一点就是，对PRBench做真实测试的时候，codex 将大量的token用于优化测试环境，这其实偏离了初衷，因为测试的目的是暴露 phycode 本身在功能上的缺陷。看到这个并及时制止是非常必要的。
